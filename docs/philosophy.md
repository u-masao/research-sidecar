# 研究プロセス自動化の思想 (Philosophy)

本ドキュメントでは、この研究開発用テンプレートプロジェクトの根底にある設計思想と、解決しようとする課題について解説します。

## 1. 背景 (Background)

データサイエンスや機械学習の研究開発において、実験コードの実装や実行、結果の集計といった「作業」に多くの時間が割かれ、本来最も重要であるはずの「思考（Why/What）」の時間が圧迫されるという課題がありました。

また、実験サイクルを高速に回す中で、以下の「ミッシングリンク」が発生しがちでした：
*   **意図の喪失:** 「なぜこの実験をやろうと思ったか」がコードの中に埋もれ、忘却される。
*   **考察の欠落:** 結果が出た瞬間の「気づき」が記録されず、後でレポートを書く際に思い出せない。
*   **再現性の低下:** どのコード（Commit ID）がどの結果（Metrics）を生んだかの対応関係が不明瞭になる。
*   **リポジトリの肥大化:** 実験ごとのログ、モデルの重み、一時ファイルなどがメインブランチに混入し、ソースコードの管理が煩雑になる。

## 2. 目的 (Purpose)

本テンプレートの目的は、**「思考（Spec）」から「考察（Reflection）」までのサイクルをシステム的に強制・自動化すること**です。
AIエディタ（Antigravity, Cursor等）との協働を前提とし、人間が「考えること」だけに集中できる環境を提供します。

## 3. キーコンセプト (Key Concepts)

本システムは以下の柱で構成されています。

### 3.1. Spec-First Workflow (仕様書駆動)
コードを書く前に、必ず「実験仕様書（`spec.md`）」を作成します。これは実験の「意図」を明確にするアンカー（錨）の役割を果たします。AIエディタはこのファイルを読み込むことで、的確なコード生成が可能になります。

### 3.2. Forced Reflection (強制振り返り)
実験が終了した直後、システムが自動的に「振り返りモード」に移行します。AIエージェントが結果（`metrics.json`）を読み取り、「仮説とどう違ったか？」と人間にインタビューを行います。これにより、熱い状態のままの思考が `reflection.md` に記録されます。

### 3.3. Zero-Shot Reporting (レポート自動生成)
`spec.md`（背景・手法）、`metrics.json`（結果）、`reflection.md`（考察）という構造化されたデータが揃っているため、レポートや論文のドラフトをいつでも即座に生成（Zero-Shot Generation）できます。

### 3.4. Sidecar Pattern (実験と実装の分離)
「ソースコード（論理）」と「実験アーティファクト（事実）」は、ライフサイクルが異なります。
*   **Source Code:** 継続的に改善され、バージョン管理されるべき資産。
*   **Experiment Artifacts:** その時点でのスナップショットであり、不変であるべき事実。

本システムでは、実験実行環境をメインのソースコードツリーから分離する「サイドカーパターン」を採用しています。これにより、メインブランチを常にクリーンに保ちながら、過去の実験結果を確実に保存・参照することを可能にします。
